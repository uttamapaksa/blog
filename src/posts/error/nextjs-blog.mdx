#### 국문/영문 폰드 별도 적용 ❌ <small className="text-gray-500">24.11.05.</small>
`@font-face`의 `unicode-range`를 사용하여 한글에만 나눔바른고딕을 적용하였지만, 강제 새로고침 시에 레이아웃 시프트(FOUT/FOIT)가 발생하였다. 주석처리하고 Next.js의 글꼴 관리 기능인 `next/font`로 변경하였다. `next/font`의 `localFont`는 빌드 시점에 글꼴을 다운로드하여 포함시키고 자동으로 최적화 해주기 때문에 레이턴시가 발생하지 않는다. `global.css`는 전역 스타일 시트로서 body의 css변수를 참조할 수 있게 되어있다. `layout.tsx`에서 `localFont`로 글꼴을 선언하고 body에 클래스로 집어넣으면 `global.css`에서 CSS 변수로 사용할 수 있다.

#### Github Pages 동적 라우팅 ⚠️ <small className="text-gray-500">24.11.08.</small>
Github Pages는 SSG만 지원하기 때문에 dynamic routing이 불가능하다. 그래서 모든 페이지를 미리 다 빌드해놓아야 한다. 검색해보면 page router와 `getStaticProps`를 쓰는 경우가 대부분인데 나는 app router로 구현했기 때문에 `generateStaticParams`를 사용하였다. `generateStaticParams`를 이용하면 설정한 경로들에 대한 data fetching까지 빌드 시에 모두 이루어진다! `npm run bulid`를 하고 콘솔을 찍어보면 알 수 있다. 다만 배포 과정에서 자꾸 오류가 발생해서 마무리하진 못했다. 정적 파일로 내보내기, sharp 모듈 오류, 이미지 최적화, 다크모드 등 여러 면에서 번거로운 점이 많았다. 결국 Github Blog를 포기하고 Vercel로 배포 환경을 옮겼다.

#### MDX 내용 스타일 적용 ✅ <small className="text-gray-500">24.11.09.</small>
`.mdx` 파일을 파싱해서 HTML로 변환을 마쳤는데 스타일이 적용되지 않았다. Network 탭에선 잘 받았는데 화면에서는 마크다운 태그들이 순수 문자열로 출력됐다. 서버 컴포넌트에서 파라미터에 맞는 폴더명과 파일 내용을 `fs` 모듈로 비동기로 다 읽어오면, `next-mdx-remote` 라이브러리를 통해 frontmatter와 content로 나누고 HTML로 변환해서 Page 컴포넌트까지 잘 전달했다. 문제는 HTML을 받는 엘리먼트에 Tailwind의 `prose` 클래스를 넣고 `tailwind typography` 플러그인을 다운받아야 했다. 이 부분을 놓쳐 하루를 통째로 날렸다.

#### 다크모드 플리커링 ✅ <small className="text-gray-500">24.11.12.</small>
FOUC(Flash of Unstyled Content) 문제. 다크모드에서 새로고침을 하면 잠깐 동안 라이트모드였다가 다크모드로 바뀌어 화면이 깜빡이는 문제다. 렌더링이 완료된 후에 자바스크립트로 테마 스타일이 적용되면 발생한다. `dangerouslySetInnerHTML`라는 인라인 스크립트를 활용했다. html에 다크모드 적용 스크립트 태그를 삽입하면 HTML 파싱을 멈추고 자바스크립트가 실행되어 렌더트리가 완성되기 전에 body에 다크모드 클래스를 추가하는 원리이다. 플리커링 방지만을 목적으로 한다면 쿠키에 테마 정보를 저장하고 서버 사이드에서 body에 클래스를 추가해 CSSOM 트리에 미리 반영되도록 하는 방식이 훨씬 간편하고, hydration 오류도 발생하지 않는다. 하지만 `window.matchMedia`를 통해 시스템 설정을 미리 읽어오는 것은 브라우저 클라이언트측 API라서 쿠키로는 첫 방문 깜빡임을 해결할 수 없다. 추가로 `tailwind.config.ts`에 `darkMode: 'selector'`를 설정해야 Tailwind가 dark 클래스를 감지하고 스타일을 적용할 수 있다.

#### 이미지 최적화 ⚠️ <small className="text-gray-500">24.11.16.</small>
이미지 최적화의 핵심 중 하나는 크기 조정이다. 성능 저하와 레이아웃 시프트를 방지할 수 있다. 가장 편한 방법은 local image를 static import하는 것이다. Next.js의 `<Image />` 태그는 자동으로 크기를 조정해준다. 추가로 기존의 .jpg 파일을 모두 .webp로 변환해서 저장했다. 크기는 모두 fill 속성으로 통일했다. 반응형 sizes 속성, remote image 최적화, device size에 따른 최적화 등 아직 개선의 여지가 많다.

#### 내비게이션 CSS 하이라이트 ✅ <small className="text-gray-500">24.12.13.</small>
내비게이션 현재 메뉴에 bottom border로 하이라이트를 주었다. hover 시에는 다른 색깔을 준다. border 두께만큼 글자가 위로 올라오는 문제가 있었다. border를 늘 유지하고 transparent를 기본값으로 하여 글자 위치가 고정시켰다. 그런데 hover된 메뉴 이름이 올라오는 것도 강조 효과로 좋은 거 같아서 되돌렸다.

#### Link 태그 중첩 오류 ✅ <small className="text-gray-500">24.12.22.</small>
클릭 요소 안에 다른 클릭 요소를 배치하는 것은 아주 흔하다. 포스트 목록 안에 카테고리 버튼을 넣어야 했다. 그러나 `Link` 태그 중첩은 이벤트 중첩으로서 권장되지 않는다. 부모는 `Link` 태그를 사용하고 자식은 클라이언트 컴포넌트로 변경 후 `useRouter`를 사용했다. `stopPropagation()`는 이벤트 전파를 막지만 기본 동작을 막지는 못하기 때문에 부모의 기본 동작(페이지 이동)이 여전히 작동한다. `preventDefault()`를 사용해야 한다.
<br/>

#### 카테고리에 따른 포스트 필터링 ✅ <small className="text-gray-500">24.12.23.</small>
메뉴 페이지에서 카테고리 별로 포스트를 필터링 할 수 있게 변경했다. 서버 컴포넌트로 구현하려 했는데 필터링 기능을 위해선 필터링을 위한 클라이언트 컴포넌트를 새로 만들어야 했다. 서버 컴포넌트에서 페칭해놓은 포스트들을 props로 보냈다. 카테고리만 모아놓는 페이지는 데이터 페칭 함수가 달라서 정적 라우팅으로 별도로 만들었다. 쿼리 스트링을 읽어 필터링을 하고, 초기화와 clean URL을 위해 `window.history.pushState()`를 사용했다. 
<br/>

#### useState 무한 렌더링 ✅ <small className="text-gray-500">24.12.25.</small>
카테고리에 따른 포스트 필터링 과정에서 발생했다. 컴포넌트 내에서 그냥 상태를 업데이트하면, 갱신 → 렌더링 → 다시 setter 호출을 해서 무한히 렌더링된다. `new Set()`의 참조값 변경 때문인 줄 알았는데, setter 호출만 하면 (상태가 바뀌지 않아서 DOM 업데이트를 건너뛰더라도) 무조건 다시 렌더링이 된단다. state는 immutable 권장이라서 새 객체를 생성하는 건 맞았다. 상태 업데이트 조건식을 `useState` 초기값으로 옮겨서 해결했다. 이 초기값 계산식은 함수 형태로 넣어야 좋다. 계산식 자체를 넣으면 렌더링마다 (상태 업데이트는 되지 않지만) 다시 계산이 된다. 그런데 같은 페이지 내에서 클릭 시 재차 업데이트가 안돼서 `useEffect`로 바꿨다.
<br/>

#### useSearchParams 빌드 오류 ✅ <small className="text-gray-500">24.12.26.</small>
정적 렌더링을 하는 경로에서 `useSearchParams`를 사용하는 자식 클라이언트 컴포넌트를 `<Suspense/>`로 감싸지 않았다고 빌드 오류가 났다. `<Suspense/>`로 경계를 한정해주지 않으면  부모 컴포넌트까지 거슬러 올라가 동적 렌더링이 일어나기 때문이다. 정적 렌더링을 유지하고자 하는 페이지가 아니라면 (동적 서버 컴포넌트라면) 감싸주지 않아도 된다.
<br/>

#### map 반환값 문법 오류 ✅ <small className="text-gray-500">24.12.27.</small>
map 메서드를 사용해서 반복문으로 JSX 표현식을 반환하는 경우가 굉장히 많다. 이때 중괄호가 아닌 소괄호로 써야 한다. 중괄호는 코드 블록으로 인식되기 때문이다. 중괄호는 작업을 수행하는 문(Statement)을 묶고, 소괄호는 값을 반환하는 표현식(Expression)을 묶는다.
<br/>

#### notFound() 다크모드 초기화 ⚠️ <small className="text-gray-500">24.12.28.</small>
Next.js에서 `notFound()`를 호출하거나 에러가 감지되면 에러 페이지로 이동하는데, 이때 다크모드가 풀린다. 깃허브 이슈를 찾아보니 아직 해결이 안 된 것 같다. 신기하게도 `<NotFound />`를 명시적으로 반환하면 문제가 안 생긴다. 현재 다크모드를 인라인 스크립트로 적용해서 hydration error가 발생하는데, 서버 컴포넌트에서는 클라이언트에서 설정한 다크모드 정보가 없기 때문에 발생하는 것이다. 아마 이것과 관련해서 동기화 문제가 생기는 것 같다. 해결 방법은 hydration error가 안 생기도록 쿠키를 사용하면 된다. 그치만 에러를 무시하고  간단하게 `useEffect`로 해결했다.
<br/>

#### Link 태그 경로 누적 오류 ✅ <small className="text-gray-500">24.12.29.</small>
`<Link />`, `<a />`의 href나 `<img />`의 src에서 절대 경로를 사용하려면 맨 앞에 루트 디렉토리를 나타내는 `/`를 붙여줘야 한다. `/`로 시작하지 않으면 상대 경로로 인식해 경로가 누적된다. 브라우저 주소창의 경우 full absolute URLs을 적어야 하지만 HTML 문서의 경우 브라우저가 문서의 경로 context를 갖고 있기 때문에 relative URLs를 쓸 수 있다.
<br/>

#### 로컬 내비게이션 Disclosure와 주소창 동기화 ⚠️ <small className="text-gray-500">25.01.01.</small>
사이드바에 디스클로저 버튼으로 트리 형태의 로컬 내비게이션을 만들었다. 1) 각 노드를 클릭하면 여닫기가 되면서 페이지 내용이 변경되고 2) 주소창으로 접근했을 때도 경로의 노드들이 모두 열리도록 했다. 두 가지 난점이 있었다. 첫째는 Headless UI의 컴포넌트에서 여닫기 상태를 제어할 수 없었다. 'Render Props Pattern'과 'Compound Component Pattern'을 활용해 Custom UI를 만들어서 Headless UI와 유사하면서도 setter에 접근할 수 있도록 변경했다. 둘째는 URL이 바뀔 때마다 노드들이 전부 전체 리렌더링되어 상태가 초기화되었다. `[[...slugs]]` 경로에 있어서 발생한 문제였고 상위 폴더의 `layout.tsx`에 옮겼더니 해결됐다. 다만 close된 컴포넌트는 여전히 언마운트 되었기 때문에 close시에 `null`이 아니라 `display: none`을 주었다. 라우트에 포함되는 노드들은 flag 처리해 처음 한 번만 열리도록 했다. 그렇지 않으면 영원히 닫을 수 없다. 다 만들고 나니 기존 라이브러리의 `defaultOpen`을 사용한 것과 차이가 없었다. 정말이지 오버엔지니어링이 따로 없었다.
<br/>

#### 코드 블록 컴포넌트 ✅ <small className="text-gray-500">25.01.05.</small>
`next-mdx-remote`의 `compileMDX` 함수를 활용해, 문자열을 넣으면 코드 블록으로 반환하는 컴포넌트를 만들었다. `compileMDX`는 비동기 함수라서 동기적으로 렌더링되는 리액트 클라이언트 컴포넌트에서는 오류가 난다. `useEffect`안에서 해야 안전하게 렌더링할 수 있다. 즉시 실행 함수로 만들어서 `useState`에 값을 넣었다. 멀티 라인이니까 백틱 3개(\`\`\`)에 이스케이프(특수 문자를 일반 문자처럼 쓰는 것)를 다 붙여야했다.
<br/>

#### 테일윈드 동적 스타일링 미적용 ✅ <small className="text-gray-500">25.01.08.</small>
Tailwind CSS는 빌드 시점에 정적 클래스를 CSS로 변환하는데, 동적 클래스는 해당 시점에 값을 알 수 없기 때문에 처리하지 못한다. Styled-Components 같은 CSS-in-JS만 쓰다보니 동적 스타일링이 당연한 줄 알았다. `tailwind.config.ts`에 `content` 속성에 경로를 넣어주어야 한다. 그러면 Tailwind는 JIT 모드라고 해서 작성된 경로들의 모든 클래스들을 추척하고 런타임에 실시간으로 클래스를 생성해 페이지에 반영한다.
<br/>
