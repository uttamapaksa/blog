#### 브라우저의 동작 원리
브라우저 주소창에 검색어를 입력하면 검색 엔진(Google)으로 보내고 URL을 입력하면 브라우저는 DNS 서버를 조회해 URL에 맞는 IP 주소를 받아와 해당 서버와 3-way handshake로 TCP 연결을 수립한다. 그 후 서버에 HTTP 요청을 보내 HTML, CSS, JS를 응답을 받는다. HTML을 파싱해 DOM을 만들고 CSS를 파싱해 CSSOM을 만들고 중간에 script 태그를 만나면 다운받아 JS를 실행한다. DOM과 CSSOM을 조합해 렌더 트리를 만들고 크기와 위치를 계산하는 Layout, 요소를 픽셀로 그리는 Painting, 최적화면 화면을 표시하면 Compositing을 거쳐 화면에 UI가 나타난다.

#### CORS (Cross-Origin Resource Sharing)
웹 브라우저에서 보안상의 이유로 적용하는 SOP(Same-Origin Policy, 동일 출처 정책) 제한을 우회하기 위한 방법이다. SOP는 다른 출처의 스크립트가 현재 웹페이지의 리소스에 접근하지 못하도록 차단하는 정책으로, 주로 API 요청 시 적용된다. 서버가 응답에 Access-Control-Allow-Origin 헤더를 포함하지 않으면, 브라우저가 응답을 차단하여 프론트엔드 코드에서 사용할 수 없다. 이 때, 서버에서 CORS 설정을 해주어야 한다.
<hr />

#### CSR, SSR, SSG
CSR(Client-Side Rendering): 초기 HTML은 최소한의 내용만 포함되어 있고, 브라우저에서 JS를 실행하여 데이터를 가져와 DOM을 동적으로 생성하는 방식이다. 첫 로딩 속도는 초기 JS 실행 부담으로 느릴 수 있지만 SPA의 페이지 전환이 빠르고 사용자 상호작용이 많은 페이지에서 유리하다. SEO에는 불리하다.  
SSR(Server-Side Rendering): 서버에서 미리 HTML을 완성해서 응답하는 방식이다. 서버 부하가 더 들지만 초기 로딩 속도가 빠르고 SEO에 유리하다.  
SSG(Static Site Generation): 빌드 시(배포 전)에 미리 HTML을 생성하고, 이를 정적 파일로 제공하는 방식이다. 동적 데이터 반영이 어렵고 데이터 변경 시 재빌드를 해야 하지만 가장 빠른 응답 속도를 보여준다.

#### 크로스 브라우징
다양한 브라우저(Chrome, Firefox, Edge, Safari 등)와 기기(PC, 태블릿, 모바일)에서 웹사이트가 일관된 UI와 기능을 제공하도록 보장하는 기술이다. 표준을 준수하고, 브라우저별 접두어를 추가하고, CSS 초기화 작업을 하고, `Can I Use`를 통해 브라우저 지원 여부를 확인하고 `Lighthouse`를 통해 성능 및 호환성을 테스트 해보는 것이 좋다.
